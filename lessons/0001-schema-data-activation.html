Reference: database release flow

Schema, data, activation

Mission: operate a production database safely. Your first rule: structure first, data second, traffic last.

What you did

alembic upgrade head changed the schema: the tables and constraints the application needs. It did not load Knicks games.

knicksiq-load-release … --activate verified a data bundle, loaded its 101 games in a transaction, then made that one validated release the release visible to the public API.

A transaction is all-or-nothing: either all release rows appear together or none do. That is why a failed load should not make a half-archive public.

Readiness is a contract

/health/live answers “is the process running?” /health/ready answers “can this process safely serve its intended traffic?” A service can be live but not ready when its schema, data, configuration, or dependencies are incomplete.

Your next real-world check

Before running any production command, say aloud: “Which database will this touch, what will it change, and how will I verify the result?” Then run a readiness check after the change.

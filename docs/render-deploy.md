# Render Deployment

KnicksIQ ships to Render from the root `render.yaml` blueprint.

## Services

- `knicksiq-web`: static Vite frontend from `apps/web`
- `knicksiq-api`: Dockerized FastAPI backend from `Dockerfile.api`
- `knicksiq-postgres`: managed Render Postgres

The v1 production deploy intentionally keeps vector RAG disabled:

- `RAG_QDRANT_ENABLED=false`
- `RAG_HYBRID_ENABLED=false`
- `RAG_RERANKER_ENABLED=false`
- `RAG_LLM_PLANNER_ENABLED=false`

## First Deploy

1. Push the repo to GitHub.
2. In Render, create a new Blueprint and select this repository.
3. Use the root `render.yaml`.
4. Let Render create `knicksiq-api`, `knicksiq-web`, and `knicksiq-postgres`.
5. After deploy, open `/health` on the API service.
6. Open the web service and verify games load and analyst questions answer.

## Expected Startup

`SEED_ON_STARTUP=true` is set for the API service. On the first boot against a fresh
Postgres database, the API creates the schema and loads the checked-in seed data.
On later boots, it skips seeding when games already exist.

## If Render Changes Service URLs

The blueprint assumes these default service URLs:

- `https://knicksiq-api.onrender.com`
- `https://knicksiq-web.onrender.com`

If Render appends a suffix because the names are taken, update:

- `VITE_API_URL` on `knicksiq-web`
- `CORS_ORIGINS` on `knicksiq-api`

## Admin Endpoints

`ADMIN_API_KEY` is generated by Render. Public read/chat flows do not need it.
Mutation/job/report-generation endpoints require the `X-Admin-API-Key` header.

The background worker is not included in the v1 blueprint. Add it later when
online ingestion is needed.
